# 澎湖旅遊 LINE Bot 交付與交接手冊

這份文件是給非工程背景的公司接手使用，目標是讓公司知道：哪些帳號要歸公司、系統要怎麼上線、平常怎麼操作、出問題時先看什麼。

## 1. 交付原則

最重要的一點：正式營運所需的帳號與金鑰，盡量都要放在公司名下，不要長期綁在個人帳號上。這樣後續若人員異動、委外、或要換維運窗口，權限才不會卡住。

建議原則如下：

- LINE 官方帳號、LIFF、雲端主機、SMTP 信箱、API 金鑰，正式版都用公司帳號建立。
- 我可以先協助建立測試用帳號與測試環境，但正式上線前，帳號所有權最好回到公司。
- 程式碼可以交給公司 Git repo 或 ZIP 壓縮檔，但真正會影響營運的是帳號、環境變數、部署方式、備份方式。

## 2. 你這個專案真正需要交付的東西

### 必交項目

- 原始碼。
- `.env` 範例與每個欄位說明。
- LINE 官方帳號的 Channel Access Token、Channel Secret。
- LIFF ID。
- 寄信帳號與應用程式密碼。
- Mistral、Hugging Face、氣象資料 API 金鑰。
- 部署主機資訊：網址、埠號、登入方式。
- 資料維護方式：熱門行程、飯店資料、RAG 資料如何更新。

### 建議附上

- 一份「日常操作 SOP」。
- 一份「故障排除表」。
- 一份「資料更新方式」。
- 一份「權限清單」，列出哪些帳號屬於誰。

## 3. 帳號與權限怎麼安排

### 建議由公司建立的帳號

- LINE 官方帳號。
- LINE Developers 專案與 Messaging API Channel。
- LIFF 應用。
- 公司專用寄信帳號，例如 `service@company.com`。
- 雲端主機帳號，例如 Render、Railway、Cloud Run、VM、或公司既有雲平台。

### 如果你先幫忙創

可以先幫公司建測試環境，但正式交付前請做其中一種：

1. 把帳號管理員、擁有者權限轉給公司窗口。
2. 直接重建公司名下的新帳號，再把正式版切過去。

如果公司沒有工程師，我會更建議第二種或至少採取第一種的完整權限移轉，避免後續你離開後沒人能改設定。

## 4. 這套 Bot 上線時要設定哪些地方

### LINE 官方帳號

- 開啟 Messaging API。
- 設定 Webhook URL。
- 開啟 Webhook。
- 關閉自動回應或不必要的預設回覆。
- 確認 Bot 可以接收文字訊息與 postback。

### LIFF

- 建立 LIFF App。
- 填入正確的前端網址，例如 `/liff/flight` 對應的公開 HTTPS 網址。
- 把 LIFF ID 填入系統環境變數。

### 郵件

- SMTP 建議使用公司信箱，不要長期用個人 Gmail。
- 若先用 Gmail，請務必開兩步驟驗證並建立應用程式密碼。
- `NOTIFY_EMAIL` 建議設成公司窗口或客服共用信箱。

### AI 與資料 API

- Mistral API Key。
- Hugging Face Token。
- 中央氣象署 API Key。

## 5. 每日/每日常用操作

### 使用者操作

- 輸入「說明」可看功能選單。
- 輸入「真人」可切到真人客服模式。
- 輸入「結束客服」回到 AI 模式。
- 輸入「智慧查詢」可規劃行程。
- 輸入「我的收藏」可看收藏清單與詢價。

### 公司內部操作

- 每天看詢價通知信箱。
- 每天查看是否有真人客服訊息待處理。
- 不定期更新熱門行程、飯店資料、RAG 資料。

## 6. 系統要如何長開

這個專案是 Flask Web 服務，不適合只靠手動開一個視窗常駐。正式上線要放在能長時間運作的環境：

- 雲端主機或 PaaS 平台。
- 開機自動啟動。
- HTTP 對外服務最好走 HTTPS。
- 有基本的 log 檢查方式。

目前程式預設是本機 `5000` 埠啟動，正式環境通常會改成由平台注入 `PORT`，再交給反向代理或容器平台管理。

## 7. 資料怎麼新增與更新

### 熱門行程

- 編輯 `data/popular_trips.json`。
- 更新後重啟服務。

### 飯店資料

- 編輯 `data/hotels.json`。
- 更新後重啟服務。

### RAG 知識庫

- 修改 `rag/source_docs/Panghu_schedule_database.md`。
- 重新建立 FAISS 向量資料庫。
- 更新後重新部署。

### 使用者收藏與 AI 行程

- 這些資料會寫到 `storage/`。
- 這是使用者資料，不建議手動亂改，除非是在排錯或清資料。

## 8. 常見權責切分

### 交給公司的人做

- 看詢價信件。
- 回覆真人客服。
- 更新內容與行程資料。
- 決定正式公告與上線時間。

### 交給你做

- 第一次部署。
- 帳號設定協助。
- 文件與 SOP 製作。
- 重大錯誤修復或版本更新。

## 9. 建議你交付給公司的文件包

1. 原始碼與部署說明。
2. `.env` 欄位表。
3. 帳號權限清單。
4. 日常操作 SOP。
5. 故障排除 SOP。
6. 資料更新 SOP。
7. 交接聯絡人與備援聯絡方式。

## 10. 建議的實際交付流程

1. 先確認正式帳號要不要由公司重新建立。
2. 把 LINE 官方帳號、LIFF、寄信帳號、API 金鑰都整理成清單。
3. 在公司名下部署正式版。
4. 驗證：Webhook、LIFF 航班、收藏詢價、寄信、真人客服切換。
5. 做一次 30 分鐘內的操作教學。
6. 留下這份文件與緊急聯絡窗口。

## 11. 我對這個專案的交付建議

如果公司沒有工程師，最穩的做法是：

- 帳號一定要公司名義。
- 主機盡量用代管平台，不要放在你自己的電腦。
- 文件要比程式更重要，尤其是「怎麼看詢價信」、「怎麼改內容」、「出錯先找誰」。
- 若日後要多人輪值，再補管理介面或後台，不要先把第一版做太複雜。
